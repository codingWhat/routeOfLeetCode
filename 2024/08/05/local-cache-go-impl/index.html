实现高性能的本地缓存库

在日常高流量场景中(读多写少场景)，经常会使用高性能的本地缓存来应对热点流量，保障系统的稳定。本文就来揭秘下缓存库的一些”内部真相”，让我们一探究竟。

开源社区

首先我们从主流缓存库入手，了解下缓存库种类、特点以及其适用场景。在Go的生态中缓存库种类丰富，并且每个缓存都有其自己独特的”卖点”，能应对不同场景。为了方便学习和理解，我对主流库做了详细调研和源码阅读，整理出以下多维度对比图。

缓存库 社区活跃度 过期时间 GC 无GC实现原理 设计原理 淘汰机制 设计缺陷

bigcache start:7.4k,fork:591 支持 Zero GC 无指针的map, map[uint64]uint32 (分片+读写锁) + map[uint64]uint32 + fifo-buffer FIFO 不支持对key设置过期时间
fastcache start:2.1k,fork:175 不支持 Zero GC 无指针的map, map[uint64]uint32 (分片+读写锁) + map[uint64]uint64 + ringbuffer(chunks [][]byte) FIFO 不支持过期时间

offheap start:363,fork:37 不支持 Zero GC 堆外内存，syscall.Mmap 内置HashTable + syscall.Mmap - 冲突时通过本地探测法，影响性能

任何一个缓存库的诞生都是为了解决特定场景的特定问题。比如存储百万对象的map会导致GC扫描耗时变长，为了规避GC扫描的影响，这才催生了Zero GC缓存库。

综上要实现通用的高性能的本地缓存，主要围绕以下方面

  • 锁竞争。全局锁导致大量请求都在抢锁、休眠、严重影响性能
  • 数据淘汰。内存资源有限，必须要根据一定策略淘汰数据
  • GC问题。存储海量对象时，GC扫描的影响不容小觑, 就算不能无GC，也得尽可能减少扫描对象

目标

  • 高性能
  • 使用简单
  • 支持按key设置过期时间
  • 支持自动淘汰(LRU)
